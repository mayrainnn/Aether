# Database Guidelines

`aether-data` is a `sqlx`-based multi-driver data layer. It supports Postgres,
MySQL, and SQLite at the backend/repository boundary, while allowing physical
SQL to differ by driver.

## Driver Selection

Use `DatabaseDriver` and `SqlDatabaseConfig` to select a driver. Do not infer
driver behavior in application code.

```rust
// crates/aether-data/src/database.rs:13
pub enum DatabaseDriver {
    Sqlite,
    Mysql,
    Postgres,
}
```

URL schemes are normalized in one place:

```rust
// crates/aether-data/src/database.rs:28
pub fn from_database_url(url: &str) -> Option<Self> {
    let scheme = url.split_once(':')?.0.to_ascii_lowercase();
    match scheme.as_str() {
        "sqlite" => Some(Self::Sqlite),
        "mysql" | "mariadb" => Some(Self::Mysql),
        "postgres" | "postgresql" => Some(Self::Postgres),
        _ => None,
    }
}
```

`DataBackends::from_config` is the composition root. It validates config and
builds exactly the selected backend:

## Repository Query Patterns

Postgres repositories usually use static SQL constants with `$1` binds:

```rust
// crates/aether-data/src/repository/quota/postgres.rs:71
let row = sqlx::query(FIND_BY_PROVIDER_ID_SQL)
    .bind(provider_id)
    .fetch_optional(&self.pool)
    .await
    .map_postgres_err()?;
```

MySQL and SQLite use `?` binds and `QueryBuilder` for dynamic `IN` clauses and
filters:

```rust
// crates/aether-data/src/repository/auth/mysql.rs:195
let mut builder = QueryBuilder::<MySql>::new(SNAPSHOT_COLUMNS);
match key {
    AuthApiKeyLookupKey::KeyHash(key_hash) => {
        builder
            .push(" WHERE api_keys.key_hash = ")
            .push_bind(key_hash);
    }
```

When a list is empty, return `Ok(Vec::new())` before building invalid SQL:

```rust
// crates/aether-data/src/repository/quota/mysql.rs:52
if provider_ids.is_empty() {
    return Ok(Vec::new());
}
```

## Transactions

The shared transaction runner is Postgres-specific. It supports read-only and
read-write modes plus optional statement and lock timeouts.

```rust
// crates/aether-data/src/driver/postgres/tx.rs:15
pub struct PostgresTransactionOptions {
    pub mode: TransactionMode,
    pub statement_timeout_ms: Option<u64>,
    pub lock_timeout_ms: Option<u64>,
}
```

Setup statements are generated centrally:

```rust
// crates/aether-data/src/driver/postgres/tx.rs:126
pub(crate) fn build_transaction_setup_statements(
    options: PostgresTransactionOptions,
) -> Vec<String> {
    let mut statements = Vec::new();
    if options.mode == TransactionMode::ReadOnly {
        statements.push("SET TRANSACTION READ ONLY".to_string());
    }
```

Do not invent ad hoc transaction setup in repositories. Use
`PostgresTransactionRunner` when a Postgres workflow needs transaction-level
options.

## Leases

Leases are also Postgres-specific and use `FOR UPDATE SKIP LOCKED` generated by
validated SQL specs:

The runner validates owners, converts `lease_ms`, and executes the claim inside
`PostgresTransactionRunner` (`crates/aether-data/src/driver/postgres/lease.rs:89`).

## Schema and Migrations

Executable migrations stay under `migrations/{postgres,mysql,sqlite}` because
runtime code embeds them with `sqlx::migrate!`.

```rust
// crates/aether-data/src/lifecycle/migrate.rs:21
static POSTGRES_MIGRATOR: Migrator = sqlx::migrate!("./migrations/postgres");
```

Schema maintenance starts from the schema workspace:

```text
// crates/aether-data/schema/README.md:11
bash crates/aether-data/schema/compose_schema.sh generate
bash crates/aether-data/schema/compose_schema.sh compose
bash crates/aether-data/schema/compose_schema.sh check
```

`schema/README.md:31` defines the edit policy: edit `logical/*.toml` first for
portable table-shape changes, use `drivers/{postgres,mysql,sqlite}` fragments
for executable compatibility, and edit `bootstrap/postgres` for empty-database
snapshot source.

## JSON and Driver-Specific SQL

Postgres migrations may use `jsonb`; MySQL and SQLite must not. This is a real
guardrail in tests:

```rust
// crates/aether-data/src/lifecycle/migrate/tests.rs:484
fn mysql_and_sqlite_migrations_do_not_use_postgres_jsonb() {
```

At repository boundaries, prefer typed Rust structs or `serde_json::Value`. For
MySQL/SQLite, serialize JSON to strings where the physical storage is text.

## Maintenance SQL Safety

Raw SQL with identifiers must validate identifiers before formatting:

```rust
// crates/aether-data/src/backend/maintenance.rs:15
fn maintenance_identifier(value: &str) -> Result<&str, DataLayerError> {
    let valid = !value.is_empty()
        && value
            .bytes()
            .all(|byte| byte.is_ascii_alphanumeric() || byte == b'_');
```

This is why `run_table_maintenance` can safely format
`VACUUM ANALYZE "{table_name}"`.

## Anti-Patterns

DON'T share physical SQL across drivers just to reduce duplication. Share Rust
contracts and behavior; keep driver syntax explicit.

DON'T add Postgres `jsonb`, `ANY($1)`, `TO_TIMESTAMP`, or `FOR UPDATE SKIP
LOCKED` to MySQL/SQLite SQL.

DON'T edit `schema/generated/**` or executable baselines independently.

DON'T build dynamic SQL by interpolating caller strings. Use `QueryBuilder` for
values and validate identifiers before formatting.

DON'T add a repository to backend read/write registries until each selected
driver has an implementation and constructor method.
